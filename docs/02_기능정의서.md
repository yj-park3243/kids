# 기능 정의서

## 1. 인증 (Auth)

### 1.1 소셜 로그인

| 항목 | 내용 |
|------|------|
| 기능 ID | AUTH-001 |
| 설명 | 소셜 OAuth2.0을 통한 로그인/회원가입 |
| 지원 | Apple, Google (동작) / Kakao (UI 노출, SDK 미연결 — 추후 연동) |
| 트리거 | 로그인 화면에서 소셜 로그인 버튼 탭 |
| 선행 조건 | 없음 |
| 정상 플로우 | 1. 소셜 SDK 인증 → 2. 서버에 `provider` + 토큰 전송 → 3. 서버에서 소셜 사용자 정보 조회 + `SocialAccount` 매칭 → 4. 신규면 회원가입 플로우(본인인증 화면으로 이동), 기존이면 JWT 발급 → 5. 홈 이동 |
| 예외 | 인증 취소 → 로그인 화면 유지 |
| 결과 | Access Token + Refresh Token 발급, 로컬 저장 |
| Apple 비고 | `authorizationCode`를 `/auth/token`으로 교환해 `refresh_token`을 `SocialAccount`에 저장 → 회원탈퇴 시 `/auth/revoke` 호출 (App Store 5.1.1(v) 준수) |

### 1.2 이메일 로그인

| 항목 | 내용 |
|------|------|
| 기능 ID | AUTH-001-2 |
| 설명 | 이메일 + 비밀번호로 로그인/회원가입 |
| 회원가입 플로우 | 1. 이메일 입력 → 2. 이메일 중복 체크 → 3. 비밀번호 설정 (8자 이상, 영문+숫자+특수문자) → 4. 서버에 등록(`POST /auth/email/register`) → 5. JWT 발급 → 6. **본인인증(AUTH-005) 화면으로 이동** → 7. 프로필 설정 → 8. 자녀 등록 |
| 로그인 플로우 | 1. 이메일 + 비밀번호 입력 → 2. 서버 인증 → 3. JWT 발급 → 4. 홈 이동 |
| 비밀번호 찾기 | (현재 폐기) — `/auth/email/reset-password` 엔드포인트 제거, UI 미노출 |

### 1.3 회원가입 (프로필 설정)

| 항목 | 내용 |
|------|------|
| 기능 ID | AUTH-002 |
| 설명 | 본인인증 완료 후 필수 정보 입력 |
| 시점 | 본인인증(AUTH-005) 완료 직후 |
| 입력 필드 | 닉네임(필수, 2~10자, 중복체크), 지역(필수, 시/구/동 선택), 프로필사진(선택), **부모 성별(필수)**, **한부모 가정 여부(필수)**, 자기소개(선택) |
| 유효성 검사 | 닉네임: 특수문자 불가, 중복 불가 / 지역: 필수 선택 / 부모 성별: MOM·DAD 중 택1 |
| **수정 정책** | 닉네임·지역·프로필사진·자기소개: 마이페이지에서 수정 가능 / **부모 성별·한부모 여부: 가입 후 수정 불가** (운영자 문의로만 정정) |
| 결과 | 프로필 저장 → 아이 정보 입력 화면 이동 |

#### 부모 성별 / 한부모 토글 상세

| 필드 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| parentGender | enum | - | `MOM` / `DAD` (필수 선택, 수정 불가) |
| isSingleParent | boolean | false | 한부모 가정 여부 (수정 불가) |

> **노출 정책**: `isSingleParent = true`인 정보는 **한부모 전용 방(`singleParentOnly = true`)의 멤버 목록·채팅 안에서만** 배지 형태로 표시된다. 일반 방 멤버 목록, 외부 유저 프로필, 검색 결과 어디에서도 노출되지 않는다. 서버는 한부모 전용 방 진입 시에만 이 필드를 응답에 포함한다.

### 1.4 핸드폰 본인 인증 (KCP)

| 항목 | 내용 |
|------|------|
| 기능 ID | AUTH-005 |
| 설명 | KCP 본인확인 서비스로 휴대폰 실명/생년월일/CI/DI 확보 (신뢰 시스템의 핵심) |
| 시점 | **회원가입 직후 필수 단계** (소셜/이메일 가입 공통). 미인증 사용자는 메인 진입 불가 |
| 인증 업체/방식 | **KCP V2** — 사이트코드 `ALQ1Q` (match 프로젝트와 동일 키 사용) |
| 흐름 | 1. 앱이 `GET /v1/auth/kcp/form`으로 KCP 제출용 HTML 폼 수신 → 2. WebView에서 KCP 페이지 표시 → 3. KCP가 결과를 앱 커스텀 스킴(`kids://kcp-cert?...`)으로 리턴 → 4. 앱이 `POST /v1/auth/kcp/callback`으로 결과 전달 → 5. 서버가 KCP 응답 복호화·검증 후 `user.isPhoneVerified = true`, 이름/CI/DI 저장 |
| 인증 완료 시 | 유저 프로필 `isPhoneVerified = true` (어드민에서 토글로 수동 처리 가능) |
| 키 관리 | 사이트코드/암호화 키는 서버 `.env.production`에 저장. match 프로젝트와 동일 키 |

#### 복수 아이 매칭 규칙

- 방 목록 자동 필터는 **유저의 모든 자녀 개월수 중 하나라도 방의 `ageMonthMin ~ ageMonthMax` 범위에 들면 매칭**
- 방 참여 신청 시 어떤 아이를 데려갈지 선택 (복수 선택 가능, 1명 이상 필수)
- 방 멤버 목록에는 "참여 자녀" 정보만 표시 (등록한 전체 자녀가 아님)

#### 입력 필드 상세

| costDescription | string | X | 비용 설명 |
| tags | string[] | X | 최대 5개, 각 10자 이내 |

#### 자격 검증 (서버 측)

방 참여 신청 시 다음 조건을 모두 통과해야 함:

```
1. joiner.parentGender 가 room.genderFilter 와 일치 (ALL이면 통과)
2. room.singleParentOnly = true 이면 joiner.isSingleParent = true 필수
3. joiner의 자녀 중 한 명 이상이 room.ageMonthMin ~ ageMonthMax 범위 내
4. joiner.noShowCount < 3 (3회 이상은 7일 제한 / 5회 이상은 영구 제한)
5. joiner 와 room.host 가 서로 차단 관계가 아닐 것
```

#### 결과

- 방 생성, 상태 = RECRUITING
- 방장 자동 참여 (참여 인원 1명)
- 방 채팅방 자동 생성
- 번개 모임이면 동 단위 매칭 유저에게 즉시 푸시 (NEW_FLASH)

### 2.3 네이버 지도 뷰

| 항목 | 내용 |
|------|------|
| 기능 ID | ROOM-007 |
| 설명 | 방 목록을 네이버 지도 위에서 탐색 |
| SDK | Naver Map SDK (flutter_naver_map) |

#### 지도 줌 레벨별 표시

| 줌 레벨 | 표시 방식 | 동작 |
|---------|-----------|------|
| ~13 (축소/광역) | **클러스터** | 동 단위로 묶어서 원형 뱃지에 방 개수 표시. 원 크기는 개수에 비례 |
| 14~ (확대/상세) | **핀 마커** | 개별 방의 약속 장소에 핀 마커 표시 |

#### 핀 마커 탭 동작

1. 핀 탭 → 하단에 방 요약 카드 팝업 (제목, 날짜, 개월수, 인원)
2. 카드 탭 → 방 상세 화면 이동

#### 좌표 관리

- 방 생성 시 장소 주소 → 네이버 Geocoding API로 위도/경도 변환
- Room 테이블에 `latitude`, `longitude` 저장
- 지도 뷰 API: 화면 영역(SW/NE 좌표) 기반으로 해당 영역 내 방 조회

#### 클러스터 집계 API

- 서버에서 동(dong) 단위로 GROUP BY → 동별 방 개수 + 대표 좌표 반환
- 약속 날짜 지난 방은 집계에서 제외

### 2.3 방 상세 조회

| 항목 | 내용 |

#### 참여 가능 조건 체크

- 이미 참여 중이 아닌가

- 인원이 꽉 차지 않았는가
- 모집 중 상태인가
- 내 `parentGender`가 방의 `genderFilter`와 일치하는가
- 방이 한부모 전용이면 내 `isSingleParent = true`인가
- 노쇼 카운트가 제한 임계치 이하인가 (3회 미만)
- 방장과 차단 관계가 아닌가

### 2.4 방 참여 신청

| 항목 | 내용 |
|------|------|
| 기능 ID | ROOM-004 |
| 설명 | 방 참여 신청 |
| 권한 | 참여 조건 충족 유저 |

#### 자유 입장 (FREE)

1. 참여 신청 → 즉시 참여 확정
2. 방 채팅방 자동 입장
3. 방장에게 알림: "OOO님이 참여했습니다"

#### 승인 필요 (APPROVAL)

1. 참여 신청 → 상태: PENDING
2. 방장에게 알림: "OOO님이 참여를 신청했습니다"
3. 방장이 프로필 확인 후 수락/거절
4. 수락 → 참여 확정, 채팅방 입장, 신청자에게 알림
5. 거절 → 신청자에게 알림

#### 참여 신청 상태

```
PENDING → ACCEPTED / REJECTED
              ↓
          CANCELLED (본인 취소)
```

#### 한부모 정보 노출 정책

## 3. 채팅 (Chat) — NestJS WebSocket Gateway

> 기획 단계에서는 Firestore로 검토했으나, **실제 구현은 NestJS WebSocket Gateway(Socket.IO) + PostgreSQL**로 결정. Firestore/Firebase Storage는 사용하지 않음.

### 3.1 아키텍처

```
Flutter App ←(Socket.IO)→  NestJS Gateway (/chat namespace)
                                   ↓
                             PostgreSQL (chat_message)
```

- 전송/수신: Socket.IO 이벤트 (`message`, `join`, `leave` 등)
- 저장: `chat_message` 테이블 (Room.id 가 chatRoomId)
- 히스토리/목록 조회용 REST: `GET /v1/chat/rooms`, `GET /v1/chat/rooms/:roomId/messages`
- 권한: JWT 핸드셰이크 + 서버에서 `RoomMember` 소속 검증

| 매너 온도 영향 | 노쇼 1회당 매너 온도 -0.3°C |

### 10.4 표시

| 위치 | 노출 |
|------|------|
| 본인 프로필 | 정확한 카운트 (소수점 1자리) |
| 다른 유저 프로필 | "노쇼 없음 / 가끔 / 자주" 3단계 텍스트 (정확 수치 비공개) |
| 방 참여 신청 시 방장 화면 | 카운트 정확 수치 |

#### API

| 엔드포인트 | 설명 |
|------------|------|
| `POST /v1/rooms/:id/attendance` | 출석 체크 (방장) `[{userId, attended}]` |
| `GET /v1/users/me/noshow` | 본인 노쇼 카운트 + 제재 상태 |

### 12.1 가이드 콘텐츠

| 항목 | 내용 |

| 위치 | 노출 |
|------|------|

## 13. 카카오톡 공유 (Kakao Share)

### 13.1 방 공유

| 항목 | 내용 |
|------|------|
| 기능 ID | SHARE-001 |
| 설명 | 방을 카카오톡으로 친구에게 공유 |
| SDK | Kakao Flutter SDK (`kakao_flutter_sdk_share`) |
| 템플릿 | Kakao Developers 콘솔에 Custom Template 등록 — 이미지(방 대표 사진 또는 기본 이미지) + 제목 + 본문(날짜·지역) + "참여하기" 버튼 |
| 딥링크 | `kids://room/:id` — Android App Link + iOS Universal Link 설정. 미설치 시 스토어로 fallback |
| 진입 | 방 상세 우상단 공유 버튼 + 방 생성 완료 직후 "친구에게 알리기" 모달 (특히 번개 모임에서 강조) |

### 13.2 추적

| 항목 | 내용 |
|------|------|
| 클릭 추적 | 딥링크에 `?source=kakao&inviter=:userId` 쿼리 — 가입 전환 시 inviter에게 통계 노출 (어드민용) |
| 그로스 KPI | 카톡 공유 → 클릭 → 가입 → 첫 방 참여까지의 퍼널을 어드민 대시보드에서 추적 |

#### API

- 클라이언트 SDK 직접 호출 (서버 API 없음)
- 딥링크 처리는 Flutter `app_links` 패키지로 라우팅
